## 二、待修问题（按优先级）

### P0-1 worker 一回合即停，常驻循环契约在运行时不成立

- **证据**：实战 1 TASK-005 无人认领 8.5 分钟；实战 2 T2 在 TASK-008 创建前
  空转结束回合，心跳停滞 502s，永不自行返回。
- **根因**：交互式 claude 是回合制运行时——回合结束即停在提示符等输入。
  循环契约（target 文字 + cycle 动词）只能约束"回合内"行为，无法让会话
  跨回合自驱。worker-loop 压力场景通过的是**意图级验证**（agent 描述了
  正确行为），不是**运行时级验证**（运行时能否执行该行为）。
- **修复建议**：`launch-role.ps1` / `taskboard_t0.py` 生成的初始 prompt
  直接以循环驱动器开头：`/loop 2m /taskboard-dev T{N}`（Claude Code 客户端），
  使循环由客户端机制而非 agent 自觉维持。launcher 按 agent 类型选择对应的
  循环原语；无循环原语的客户端回退为 T0 按交接重新拉起。
- **验收**：真实终端 worker 在空队列时跨回合存活 ≥3 个循环周期，新任务
  入队后一个周期内被认领，全程零用户操作。

### P0-2 过期 Goal Complete 哨兵阻塞新 milestone

- **证据**：实战 2 中旧 STATE.md 的 `Goal Complete: yes` 使 `taskboard_t0.py`
  判定 complete：不输出启动命令、target 文件不随新 goal 刷新（仍为 6-10
  旧目标），T0 被迫手工改写三个 target。
- **修复建议**：当显式传入的 `--goal` 与 `goal.json`/STATE.md 哨兵所属
  milestone 不一致时，视为新 milestone 开始：复位哨兵（或提示 T1 复位）、
  强制重新生成 target、正常输出启动计划。
- **验收**：在已完成看板上用新 goal 调用 t0.py，target 内容与时间戳刷新，
  启动命令正常输出，零手工修补。

### P1-3 心跳依赖 agent 自觉，T1 整场未写

- **证据**：实战 2 中 T2/T3 写了心跳，T1 从未写（target 文字有要求），
  T0 只能靠 CPU 占用推断 T1 存活。
- **修复建议**：心跳进启动 prompt 的强制序列（与 P0-1 合并实现：loop 的
  每轮第一动作即 `taskboard.py alive T{N}` 或 cycle）；可选：launch-role.ps1
  在拉起 claude 前先代写一次初始心跳。

### P1-4 T0 的 stall 处置退路再次落到用户头上

- **证据**：实战 2 中 T0 诊断正确后，给出的首选方案是"请你到三个 tab
  各贴一条 /loop"——与既有规则"绝不让用户手动管理 T1/T2/T3"冲突
  （实战 1 已因同样行为被用户否决过一次）。
- **修复建议**：role-t0 固化 stall 响应顺序：能用客户端机制自动驱动的
  绝不转嫁用户；用户操作只允许出现在"首次启动确认页"这类产品强制点。
  P0-1 修复后此问题大部分自然消失。

### P2-5 入口可发现性：用户敲 `/taskboard_start` 报 Unknown command

- **证据**：实战 2 开场用户直觉输入 `/taskboard_start`，失败后才改用
  `/taskboard-dev T0`。
- **修复建议**：T0 ready 确认语中固定印出一键入口命令；或评估提供
  斜杠命令包装的可行性。

### P2-6 控制台中文乱码（GBK 显示层）

- **证据**：两次实战 + 本仓库 smoke 输出中文均为乱码。不影响功能，
  持续影响可读性与信任感。
- **修复建议**：控制面脚本输出前显式设置 UTF-8（或 `PYTHONIOENCODING`
  指引），或关键字段 ASCII 化。

### P2-7 遗留确认项

### P0-8 T1 冷启动死角：确定性 next 无视新 milestone 叙述

- **证据**：空看板 + target 文件中的新 milestone 指令 → T1 每轮
  `/taskboard-next` 判 idle，叙述指令被完全忽略，卡 8–10 分钟，最终靠
  强制交互式 prompt 重启解决。
- **根因**：T0 的 next 优先级里有"无活跃任务但 milestone 未完成 → 唤 T1
  建任务"，但 T1 自身的 next 没有等价分支——确定性队列没有
  "milestone-bootstrap" 状态。
- **修复建议**：`taskboard next T1` / cycle 增加 bootstrap 分支：当
  goal.json/target 的 milestone 新于看板（无对应 REQ 段或哨兵已复位）且
  队列为空时，返回 `bootstrap-new-milestone` 动作而非 idle。
- **验收**：空看板注入新 goal 后，T1 在一个循环周期内开始建
  REQUIREMENTS/TASK，零人工干预。

### P0-9 T2 空置 20 分钟自行取消 /loop

- **证据**：T2 以"看板空置 20 分钟"为由主动 cancel 自己的 /loop cron，
  时机恰逢 5 个新任务入队，卡 15–20 分钟，需用户"重启T2"救场。
- **性质**：直接违反"空队列绝不退出"规则——worker-loop 纪律再次
  意图级通过、运行时级失守。
- **修复建议**：循环退出条件收紧为机器可判定的两种：STATE.md
  goal-complete 哨兵、用户显式停止。"空置时长"不是合法退出理由；
  取消自身 loop 列入 Red Flags（"看板空了这么久，loop 没意义了"）。

### P1-10 T0 kill 撞车：未检查活动状态即重启 worker

- **证据**：用户刚把指令贴入正在 4 分钟深度分析的 T1，T0 未检查即 kill
  该进程，分析成果丢失（后靠重做恢复）。T0 已自认失误。
- **修复建议**：role-t0 固化 worker 重启前置检查：CPU 活动增量 + 声明
  意图并短暂等待；存活 worker 原则上只改 target 重新驱动，不杀进程。

### P1-11 `claude -p` headless 在实战机器上卡死

- **证据**：headless 强制重启方案卡死（PID 3 分钟仅 5.2s CPU），被迫
  改交互式。注意 spawn 探测自身依赖 `claude -p`，同机探测可能产生
  假"超时→不可用"判定。
- **修复建议**：探测与 headless 派发增加硬超时 + 失败时回退交互式路径；
  把"headless 可用性"并入启动探测的分类输出。

## 四、测试方法论修订（写入 tests/pressure/README.md）

压力场景须标注验证级别：

- **意图级**：subagent 模拟，验证 agent 是否知道并描述正确行为。
- **运行时级**：真实终端/真实客户端，验证运行时是否允许该行为发生。

worker-loop 场景是两级分裂的标本（意图级 PASS、运行时级 FAIL）。
涉及"跨回合存续、进程生命周期、客户端机制"的规则，意图级通过不得
作为落地依据。
